## 개발시 사용할 전략
저희는 Branch 전략을 사용할 것입니다. (Branch는 main(본 프로젝트 파일)이 아닌 곁가지처럼 약간의 서브 파일을 두는 것을 말합니다.  약간 부가기능을 작성한 코드라고 할까요..?)
Branch전략이란 특정 기능(맵, 건물, 재화 생성 로직 등)을 코딩 후 main에 바로 push하는 방식이 아닌 branch에서 작업한 후에 main에 합치는 방식을 말합니다.
장점으로는 main에 곧 바로 push하지 않으니 유지 보수에 좋다는 장점이 있습니다. 또 추가로 생각해보기

원래는 PR(Pull Request : 브랜치에서 작업 후 팀원끼리 작성한 코드를 검토하며 main에 push하는 방식)기반 브랜치 전략이 근본이지만 
프로젝트 진행 속도를 높이기 위해 PR 없이 branch에서 빌드 후 main에 push하는 방식을 쓰겠습니다.

관련 git 코드는 밑에 코드 블럭으로 두겠습니다.
```Bash
git branch : 현재 작업하고 있는 branch와 branch목록을 볼 수 있습니다.
push전 이 명령어를 통해 작업하고 있는 디렉토리가 main인지 특정기능을 위한 branch인지를 알 수 있습니다.
참고로 잘은 기억 안나지만 현재 작업하고 있는 브랜치는 앞에 기호 *가 붙었던 것 같습니다.

git checkout -b feature/만들 기능 제목 : 이 명령어는 브랜치를 생성하는 명령어입니다.
(작성할 브랜치명(feature/기능 제목)은 가독성을 위한 것이며 브랜치 명은 편한대로 지어도 좋습니다..!)

git checkout 브랜치명/main : 브랜치나 main으로 작업하고 있는 디렉토리를 이동합니다.
```

# push와 pull
push는 작업한 코드를 현재 작업하고 있는 워크 스페이스(브랜치나 main)에 저장합니다.
push 명령어를 쓰기 위해서는 아래의 코드처럼 몇가지 선행과정이 필요합니다.
```
git add . : 작성한 코드를 확인할 때 사용합니다. 사용시 어떤 브랜치에서 어떤 코드를 썼고, 어떤 작업을 했는지 보여줍니다.
git commit -m "커밋할 때 사용할 메시지" : 현재 작성한 코드를 기기(로컬 환경)에 임시 저장합니다. 커밋할 때 메시지를 통해 어떤 작업을 했는지 팀원에게 보여줄 수 있습니다.
git push branch명/main : branch나 main에 저장합니다. 이 코드를 통해서 팀원들에게 코드를 공유할 수 있습니다.

최신버전 가져오기
git checkout main : 현재 작업 디렉토리를 main으로 이동합니다.
git pull origin main : pull : main의 최신 코드를 가져옵니다.
처음에 프로젝트를 받을 때는 git clone 명령어를 사용하였지만, pull 명령어는 프로젝트를 다운 받은 이후에 main에서 원격으로 가져올 때 사용합니다.
```

※가끔 merge충돌이 일어날 수 있는데 이건 음.. 저도 해결 못 합니다.. 지피티랑 같이 해봐요..!
그래도 충돌이 생기는 원인을 말해보자면, main에서 최신버전을 pull을 안 하고 작업하다가 현재 작업중인 파일과 충돌이 자주 발생합니다.
예시를 들어보면, 최신 버전은 코드가 a+b까지 작성되어있고, 제가 코드 a에서 추가로 작성해서 a+c의 코드를 만들었습니다.
이 경우 b와 c가 겹치지 않는 영역이라면 merge 기능으로 살릴 수 있지만 만약 b와 c의 코드가 겹치게 된다면 상당히 골치아파집니다.

그래서 작업할 때는  가능한 main에서 pull하고 작업하시는 것을 추천드립니다.
